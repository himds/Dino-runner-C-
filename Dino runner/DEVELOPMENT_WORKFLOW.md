# Dino Runner 开发流程文档

这份文档是给组内成员看的，目标是统一大家新增功能时的做法，避免前后端不同步、数据库漏改、页面能点但接口没实现这类问题。

---

## 1. 项目结构说明

### 后端
- `Program.cs`：项目入口，负责注册服务、数据库、路由、中间件
- `Controllers/`：后端 API 接口
  - `AuthController.cs`：登录、注册
  - `UsersController.cs`：用户信息
  - `ScoresController.cs`：分数上传、排行榜、成就判定
  - `AchievementsController.cs`：成就定义、用户已解锁成就
  - `ShopController.cs`：商城商品、购买逻辑、用户已购道具
- `Data/AppDbContext.cs`：EF Core 数据库上下文和种子数据
- `Models/`：实体类
- `Dtos/`：接口请求/响应的数据结构

### 前端
- `wwwroot/login.html`：登录 / 注册页面
- `wwwroot/index.html`：游戏主页面
- `wwwroot/shop.html`：商城页面
- `wwwroot/achievements.html`：成就页面
- `wwwroot/style.css`：统一样式文件

## 2. 新增功能的标准流程

无论是新增“商城道具”“成就”“排行榜筛选”“新的页面”，都建议按这个顺序做。

### 第一步：先明确功能需求
先回答下面几个问题：
1. 这个功能是给谁用的？
2. 它出现在前端哪个页面？
3. 它需要后端接口吗？
4. 它需要数据库新增字段/表吗？
5. 它的输入和输出分别是什么？

如果这一步不明确，后面很容易返工。

---

### 第二步：先设计数据，再写页面
新增功能时，不要先乱改页面。
优先顺序应该是：

1. **设计数据结构**
2. **设计 API**
3. **实现后端**
4. **最后接前端页面**

举例：如果要新增一个“复活道具”
- 先确定数据库是否要增加道具定义
- 再确定前端怎么知道玩家拥有这个道具
- 再确定游戏页如何消费这个道具

---

## 3. 涉及数据库的功能，怎么做

如果新增功能需要持久化数据，走下面流程。

### 3.1 修改 Model
在 `Models/` 中新增实体类，或者给已有实体增加字段。

例如：
- 新增 `Type`
- 新增 `Icon`
- 新增 `Charges`

### 3.2 修改 DbContext
在 `Data/AppDbContext.cs` 中：
- 添加 `DbSet`
- 配置关联关系
- 如果有初始数据，在 `SeedData()` 中加入种子数据

### 3.3 更新数据库
当前项目使用 SQLite，并在启动时 `EnsureCreated()`。
如果只是开发阶段小改动，可以删数据库重建；
如果是正式协作，建议统一迁移方案，不要每个人都手动乱删库。

---

## 4. 涉及接口的功能，怎么做

### 4.1 先定义接口职责
先写清楚：
- 路由是什么
- 请求参数是什么
- 返回什么 JSON
- 失败时返回什么错误

例如商城购买：
- `POST /api/shop/purchase`
- 请求：`{ userId, shopItemId }`
- 返回：`{ message, coins }`

### 4.2 再写 DTO
如果接口要接收复杂参数，就放到 `Dtos/` 里。

规则：
- 请求参数用 Request DTO
- 结构复杂的响应也可以单独建 Response DTO

### 4.3 再写 Controller
Controller 里建议每个接口都做这几件事：
1. 校验用户/数据是否存在
2. 校验业务条件是否满足
3. 写数据库
4. 返回前端需要的最小数据

不要把一堆页面显示逻辑塞到 Controller 里。

---

## 5. 涉及前端页面的功能，怎么做

### 5.1 先决定页面入口
新增功能属于哪个页面：
- 登录相关 → `login.html`
- 游戏相关 → `index.html`
- 商城相关 → `shop.html`
- 成就相关 → `achievements.html`

如果是全新模块，再考虑是否新建页面。

### 5.2 前端开发顺序
建议顺序：
1. 先写 HTML 结构
2. 再写 JS 数据请求
3. 再处理状态提示
4. 最后再做样式微调

### 5.3 页面 JS 约定
每个页面建议都保留这些结构：
- `api()`：统一请求后端
- `initUser()`：读取本地登录信息
- `loadXxx()`：加载当前页面数据
- `init()`：页面入口函数

这样组员接手时比较容易看懂。

## 6. 新增一个完整功能时的推荐步骤

这里给一个通用模板。

### 示例：新增一个“新商城道具”

#### 第 1 步：需求定义
确定：
- 道具名称
- 价格
- 效果
- 在游戏中何时生效

#### 第 2 步：数据层修改
- 修改 `ShopItem` 模型（如果需要）
- 在 `AppDbContext.SeedData()` 中加入新道具

#### 第 3 步：后端接口确认
确认这些接口是否够用：
- `GET /api/shop/items`
- `GET /api/shop/user/{userId}`
- `POST /api/shop/purchase`

如果不够，再补接口。

#### 第 4 步：前端商城页接入
在 `shop.html` 中：
- 读取商品列表
- 显示价格、描述、是否拥有
- 购买成功后刷新金币和拥有状态

#### 第 5 步：前端游戏页接入
在 `index.html` 中：
- 加载已购道具
- 按道具 ID 或类型激活效果
- 处理道具消耗逻辑

#### 第 6 步：测试
至少测这几种：
- 未购买时是否不生效
- 购买后是否生效
- 金币不足时是否提示正确
- 重复购买时是否被拦截
- 游戏结束后数据是否正常

## 7. 组内协作建议

### 7.1 不要同时乱改同一个文件
像 `index.html`、`AppDbContext.cs` 这种核心文件，最好同一时间只让一个人主改。

### 7.2 改接口时必须同步前端
后端接口字段一改，前端对应字段也要一起改。

例如：
- 后端把 `coins` 改成 `totalCoins`
- 前端如果还读 `coins`，页面就会出错

### 7.3 提交代码前自查
提交前至少检查：
- 页面能不能打开
- 接口能不能返回数据
- 控制台有没有报错
- 数据库有没有脏数据

## 8. 注释规范建议

以后新增代码，建议按下面方式写注释。

### 后端注释
写在：
- Controller 类上
- 每个接口方法上
- 复杂业务判断前

例子：
```csharp
/// <summary>
/// Purchase a shop item for the current user.
/// </summary>
```

### 前端注释
写在：
- 页面初始化入口
- API 请求函数
- 游戏主循环
- 复杂状态逻辑

例子：
```javascript
// Load owned items from backend and activate effects for the current run.
```

## 9. 推荐开发顺序总结

以后新增功能，统一按这个顺序：

1. 看需求
2. 设计数据
3. 设计接口
4. 写后端
5. 写前端
6. 联调测试
7. 提交代码

不要一上来就先改页面文字或者乱加按钮，不然最后经常要返工。

## 10. 当前项目特别提醒

### 路由顺序
`Program.cs` 中应该先：
- `MapControllers()`
再：
- `MapFallbackToFile("index.html")`

否则可能导致 API 请求被前端回退路由拦住。

### 商店道具逻辑
现在游戏页使用 `/api/shop/user/{userId}` 读取已购道具。
新增道具时，前端游戏逻辑也要同步处理，不然买了也不会生效。

### 本地登录状态
当前前端使用 `localStorage.dino_user` 保存：
```json
{ "id": 1, "coins": 100 }
```
如果后续要加 token，需要统一改所有页面的登录判断逻辑。

## 11. 建议每次开发都写一个小 checklist

每次做新功能前，可以照着这个清单走：

- [ ] 需求明确
- [ ] 数据结构确认
- [ ] 接口定义完成
- [ ] 后端实现完成
- [ ] 前端接入完成
- [ ] 异常流程测试完成
- [ ] 提交前自测完成

---

如果以后项目继续扩展，建议把这份文档继续细化成：
- 前端规范
- 后端规范
- 数据库规范
- 提交流程规范

这样组员接手会轻松很多。
